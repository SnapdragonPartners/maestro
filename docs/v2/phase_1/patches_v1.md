## Patch Appetite (set 2026-07-22, ADR 0025 conformance-first amendment)

The bar for patching the target is **context-dependent, and it has now tightened.** While no story completed at all, a larger patch was justified because the alternative was zero signal — that is why P-1…P-8 run as large as they do. Now that runs mostly complete and the work is refinement, the rule is:

- **Patch** when the fix is small: order of ten lines, confined to a couple of functions.
- **Otherwise record the diagnosis and park the rung** as blocked-on-v2. A permanently red rung still earns its place if its diagnosis is written down — under the conformance framing, the diagnosis *is* the yield.
- **Scope the decision by the shape that makes the patch sound, not by its first diff.** P-11 began as ~15 lines in one function and was only correct once every writer to the resource took the lock — three packages, seven call sites, a new exported API and test suite. A fix that needs a resource-wide audit to be correct is a deliberate defer-or-commit call at the moment of discovery, not something to drift into three review rounds deep.

Patches to the v1 path remain instrument work, never v1 maintenance, and are never backported to `v1-freeze` — v1 is scaffolding that dies on schedule, so the appetite shrinks further as v2 becomes the target.

Adapter-side consequences (benchmark module, not v1 patches): the v1 adapter now declares `streamed` enforcement, tails the usage log (deltas via `ReportUsage`, engine-cancelled at caps), takes the log as the canonical tokens/cost/`llm_calls` source, validates the surface version on both handshake halves, and exports `usage.jsonl` as evidence.

**Dependency-API exploration (found 2026-07-22, not patched).** On a refactor that turns on an external dependency's API, the coder does not reach for the idiomatic inspection tool. This is a **strategy gap, not a tooling gap** — an earlier draft of this entry called it a missing affordance, which was wrong: `go doc` was available via the existing `shell` tool and works in the exact coder image as uid 1000 (verified — `go doc github.com/SnapdragonPartners/maestro-llms/llms.ChatClient` returns the full interface and its design commentary in one call). `cleanup-provider-options` needed `llms.ChatClient` and the `ListModels` signature; the coder issued **111 shell calls, 101 of them distinct** — wandering rather than cycling — including 31 hand-greps into the Go module cache across two guessed `GOMODCACHE` paths, and used **`go doc` zero times**. It burned 3.09M tokens in 633s (~4.9k tok/sec against ~2.1k in healthy runs) and hit the token cap without converging. The architect diagnosed it correctly and repeatedly, so the *review* seat worked — the gap is the coder's affordance, not its supervision. `go doc` appears nowhere in any coder template or tool description, so the model is left to invent an approach and invents a bad one. The fix is therefore **cheap and in-appetite**: a prompt line pointing at `go doc` for dependency APIs — the P-4/P-5 shape, generally correct for any Go codebase rather than benchmark special-casing. Not yet applied; it would be P-12.

Remaining known candidates from v1's dying-defect list (patched only if runs hit them): the architect spec-review *strictness* defect (terse story prompts rejected on content — still has NOT surfaced; specs sail through review). The related fragilities that DID surface on frontier validation were mechanical, not strictness: the maintenance-signal mismatch (P-9) and the spec-review iteration cap (P-10). Watchdog requeue race (#221) remains WONTFIX-in-v1.
